# Управление группами

**URL:** `/admin/groups`

Группы упрощают массовое назначение прав в пространствах.

## Зачем нужны группы

```
Группа "platform-engineers"
  ├── user1@company.com
  ├── user2@company.com
  └── user3@company.com

Пространство "Platform Docs"
  └── Группа "platform-engineers" → роль editor
```

Все три пользователя получают роль editor в пространстве без индивидуального назначения.

## Создание группы

| Поле | Обязательно | Описание |
|------|:-----------:|----------|
| Название | ✅ | Уникальное имя группы |
| Описание | ❌ | Краткое описание |

## Управление участниками

1. Откройте группу
2. **Добавить участника** — выберите пользователя
3. **Удалить** — уберите пользователя из группы

## Назначение группы в пространство

**Пространства** → редактирование → **Группы**:

1. Выберите группу
2. Укажите роль (`viewer`, `editor`, `admin`)
3. **Добавить**

## Синхронизация из OIDC

При включённом `sync_groups` в OIDC-провайдере:

1. При каждом входе группы пользователя синхронизируются из JWT claim
2. Пользователь автоматически добавляется в соответствующие группы TreePage
3. Права в пространствах применяются через назначенные группы

### Настройка

**OIDC-провайдеры** → `group_claim: groups`, `sync_groups: true`

На стороне OIDC-провайдера (Keycloak пример):

1. Создайте группы: `developers`, `ops-team`
2. Добавьте mapper для claim `groups` в access token
3. В TreePage создайте группы с теми же именами
4. Назначьте группы в пространства

## Удаление группы

Удаление группы не удаляет пользователей — только связь группы с пространствами и membership.

## Примеры

### Команда разработки

```
Группа: backend-team
Пространство: api-docs → editor
Пространство: architecture → viewer
```

### DevOps

```
Группа: ops-team
Пространство: runbooks → editor
Пространство: all-docs → viewer
```
